# Phase 02: Local browser review workbench

## Goal

- Add the local browser workbench that serves the review queue, preview images,
  candidate details, and autosave endpoints.

## Scope

- `review-crops` starts a localhost HTTP server by default.
- Serve a static single-page UI with left queue, center source-image view, and
  right text editor.
- Serve JSON API endpoints for state load/save and draft updates.
- Preserve preview image paths and source provenance without copying official
  image artifacts into Git-tracked locations.

## Non-goals

- No polished production app.
- No external frontend build pipeline.
- No authentication; server is intended for local-only use.

## Affected files

- `src/leet_practice/verification.py`
- `src/leet_practice/cli.py`
- `tests/test_verification_workbench.py`
- `dev-docs/plan/issue_1/02-workbench.md`

## Implementation steps

- Add local HTTP server wrapper.
- Add static HTML/CSS/JS payload generated by Python.
- Add endpoints for queue, candidate detail, image serving, and save/update.
- Add tests for endpoint responses and state persistence.
- Document the local-only server behavior in command help.

## Acceptance criteria

- A reviewer can open a local URL for a suggestion run.
- UI exposes statuses: `unreviewed`, `accepted`, `needs_fix`, `rejected`.
- Raw OCR/provenance is separate from editable verified text fields.
- Saving progress updates `crop-review-state.json`.

## Validation commands

- `uv run --extra dev python -m pytest`
- If uv is unavailable, `python -m pytest`

## Manual smoke tests

- Start the command with `--no-open` or equivalent against a fixture and fetch
  the root page/API endpoint.

## Rollback risks

- Medium risk: long-running server commands can block tests; server startup must
  be separable from request handling for tests.
- Keep image serving constrained to files referenced by the suggestion run.
